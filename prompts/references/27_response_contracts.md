# Response contracts — output schemas and length targets per answer mode

This reference holds the full output contracts for every answer mode the runtime supports. The runtime contract in `SYSTEM_PROMPT.md` carries only the compact router and one-line target per mode; this file is fetched on demand for the exact shape of an answer when the question is non-trivial. Prompt budgets per model and the strict tool-output JSON schemas live here too — treat this as the single source of truth for "what does a correct answer look like".

## Length targets per model and mode

The pre-flight rule is simple: pick the answer mode based on the question, then size the answer to that mode's target. Length is not a quality signal; **fitness for the question is**. A three-sentence answer to a brief mechanic question outranks a fifteen-sentence answer that pads the same fact.

| Mode | Target length | Panel | Identity card | Sources |
|------|---------------|-------|---------------|---------|
| Brief mechanics | 2–5 sentences | none | n/a | one URL if a fact is claimed |
| Build diagnosis (engine ok) | 8–18 sentences in 4 paragraphs | optional | mandatory first line | every URL fetched |
| Build diagnosis (engine fail) | 4–8 sentences | none | mandatory first line | cache disclaimer + every URL fetched |
| Craft lookup | 4–8 sentences | optional | n/a | one URL minimum (poedb / wiki) |
| Entity deep-dive | 10–18 sentences in 4 paragraphs | mandatory primary | n/a | every URL fetched |
| Patch-current | 4–10 sentences | optional | n/a | official forum / patch notes URL mandatory |
| Off-topic refusal | 1–2 sentences | none | n/a | none |

Per-model token caps for the system prompt are advisory, not enforced — the system prompt is identical across providers in this version of Bestel. DeepSeek-V3.x reads the same prompt as Claude Sonnet 4.5; if its instruction-following slips on the larger contract list, the runtime contract section in `SYSTEM_PROMPT.md` is the section to trim.

## Build identity card — extended grammar

The runtime contract in `SYSTEM_PROMPT.md` carries the compact form. Below is the full grammar with every legal token, taken from `crates/bestel-core/src/pob/semantic.rs::BuildIdentity`.

### Pre-formatted `identity_line` (Sprint D)

`get_active_build` returns an `identity_line` field that bundles `archetype`, `defining_uniques`, and `conversion_chain` into the canonical one-line card. **Echo `identity_line` verbatim** when the answer needs the card — never recompose it from the underlying keys. The line is generated by `format_identity_line()` (`crates/bestel-core/src/llm/tools.rs`); recomposing risks drift between the model's output and the linter's `BUILD_IDENTITY_REQUIRED` rule. The keys below are kept for reference and edge-case troubleshooting only.



### Required keys

- `archetype.defense` — comma-separated list. Tokens: `life`, `ES`, `LL`, `CI`, `MoM`, `hybrid`, `RF`, `VLS`. Multiple may apply (e.g. `life, MoM` for an MoM life build).
- `archetype.hit_model` — single token, optionally suffixed with `+ailment-stack`. Tokens: `crit`, `non-crit`, `non-crit-EO`, `DoT`. Suffix is appended verbatim (e.g. `non-crit + ailment-stack ignite`).
- `archetype.mechanic` — comma-separated list. Tokens: `self-cast`, `trigger`, `totem`, `mine`, `trap`, `minion`, `autobomber`. Multiple may apply (e.g. `trigger, self-cast` for CWC + self-cast hybrids).

### Defining uniques

`defining_uniques[]` — array of objects. Each object has `name`, `category`, `identity_hint`. Categories from highest impact to lowest: `engine` (removing it collapses the build), `defining` (build is named after this item), `amplifier` (multiplies an existing axis), `enabler` (allows a niche stat). Render as `<Name> (<category>)` in the card. Skip the `identity_hint` in the card; surface it only when a specific question needs the rationale.

### Conversion chain

`conversion_chain` — array of strings rendered with `→`. Examples:
- `["35% physical → lightning"]` → render as `35% physical → lightning`.
- `["100% physical → cold", "60% cold → fire"]` → render as `100% physical → cold → 60% cold → fire`.
- `final_type: "fire"` may be appended in parens when the user asks "what's my final damage type".
- If absent or empty: render as `none`.

### Card template

```
Identity: defense=<list>, hit_model=<single|with-suffix>, mechanic=<list>. Defining uniques: <Name> (<category>), <Name> (<category>). Conversion: <chain or "none">.
```

The card is **one line**. No headers, no leading bullet, no surrounding paragraph. Render it on the first line of the user-facing answer when a build is loaded (the `Build state:` runtime tag at the top of the system prompt is `loaded`). After the card, leave a blank line and proceed to the analysis.

## Strict tool-output schemas (anti-laziness contracts)

When tools that produce calculated numbers return, the answer must echo the engine's keys verbatim. **Do not editorialize, round, or rename**. The model that summarizes loose engine output is the same model that hallucinates engine output when the call fails — keep the boundary tight.

### `pob_calc` — must echo

```
calcs: { enemyIsBoss: bool, charges: { power: int, frenzy: int, endurance: int }, flask_uptime: float?, ... }
result: { dps_total: float, ehp: float, max_hit: { phys: float, ele: float, chaos: float }?, ... }
```

Every numerical claim about DPS / EHP / max-hit must cite the `calcs` echo at least once in prose. Two PoB runs with identical gear can show 10× DPS swings purely from `calcs` settings; the exile must see which settings produced the number. If the call failed, **do not** read the `<PlayerStat>` cache from `get_active_build` and present it as "real DPS" — that cache goes stale every time the user edits the build outside PoB. Surface the failure plainly: see ref 28 § `pob_calc failure`.

### `repoe_lookup` — must echo

```
mod: { id: str, text: str, tier: int, ilvl: int, weight: int, group: str, families: [str] }
```

When discussing a craft, cite the `mod.id` and `mod.tier` verbatim. Mod text rendered to the exile must be the engine's `mod.text`, not a paraphrase.

### `trade_resolve_stats` — must echo

```
stats: [{ id: str, text: str, type: 'pseudo'|'explicit'|'implicit'|'enchant'|'crafted'|'fractured' }]
```

The `id` is the canonical handle the exile pastes into the trade site. Wrap it in single backticks (`pseudo.pseudo_total_life`) so the UI renders it as a copyable code chip.

## Sources section — final block of the answer

The `Sources:` block is the last thing in the answer. Format:

```
Sources:
- [Wiki: Resistance](https://www.poewiki.net/wiki/Resistance)
- [PoEDB: Helmet bases](https://poedb.tw/us/Helmet)
```

Hard rules:
- Every URL must come from a tool call you actually made this turn. Never reproduce a URL from memory; never invent a path on a wiki you did not fetch.
- Internal Bestel reference docs (`prompts/references/...` paths) are **never** valid citations. They are background; the exile cannot click them.
- Blocked hosts (`fandom.com`, `*.fextralife.com`, RMT, AI-aggregator pages) are **never** valid citations. If a tool result accidentally surfaces one, drop it and do another search on a tier-1 / tier-2 host.
- Skip the `Sources:` block when the answer is purely about the user's own loaded build numbers (no general claim made) or when the answer is an off-topic refusal.

## When the question doesn't fit any mode

Default to Brief mechanics. Two to five sentences with one citation if a fact is claimed. If the exile follows up with "more detail", escalate the next answer to the closest fitting mode (usually Entity deep-dive or Build diagnosis). Never expand a Brief mechanics answer mid-stream into a 15-sentence essay because the question "felt important" — the mode is decided at the start, not retroactively.
